# DynamoDB — Interview Questions

**What is the difference between DynamoDB partition key and sort key?**
The partition key (hash key) determines which physical partition stores the item — DynamoDB hashes the key value to assign storage. All items with the same partition key are in the same partition. The sort key (range key) orders items within a partition and enables range queries. With only a partition key, each item must have a unique key — one item per partition key value. With partition + sort key, multiple items can share the same partition key (differentiated by sort key). Example: `userId` as partition key, `orderId` as sort key lets you store many orders per user and query all orders for a user with a single Query operation.

**What is a Hot Partition problem in DynamoDB?**
DynamoDB distributes data across partitions based on partition key hash. If many requests target the same partition key value, that partition gets all the load — a "hot partition." Example: using `date` as partition key means all today's writes hit one partition. DynamoDB limits each partition to 3000 RCU and 1000 WCU — exceeding causes throttling. Solutions: add random suffix to partition key (user-id-1 through user-id-10), then query multiple partition keys and combine results. Use write sharding for high-cardinality access. Choose partition keys with high cardinality and even access distribution (userId is better than country).

**What is the difference between a GSI and an LSI?**
LSI (Local Secondary Index) uses the same partition key as the table but a different sort key. Must be created at table creation (cannot add later). Shares throughput with the table. Up to 5 LSIs per table. Use when you need alternate sort orders within the same partition. GSI (Global Secondary Index) can have completely different partition and sort keys. Can be added to existing tables. Has its own throughput settings. Up to 20 GSIs per table. Use for completely different access patterns. Choose GSI when you need to query on non-key attributes; choose LSI when you need multiple sort orders for the same partition key.

**What is DynamoDB Streams and what are common use cases?**
DynamoDB Streams captures a time-ordered sequence of item-level changes in a table. Each change (insert, update, delete) creates a stream record with the old and/or new image of the item. Stream records are available for 24 hours. Integration with Lambda (event source mapping) enables processing every change. Common use cases: replication (replicate to another region for DR or global tables); aggregation (maintain counts, totals in another table); search (sync changes to Elasticsearch for full-text search); audit logging (track every change with before/after values); event-driven architecture (trigger workflows on data changes).

**How do you perform a migration from RDS to DynamoDB?**
First verify DynamoDB fits your access patterns — if you need complex queries, joins, or ad-hoc analytics, reconsider. Design the DynamoDB schema around your access patterns (single table design). Use AWS Database Migration Service (DMS) for the initial data load and ongoing replication during cutover. Or write a migration script that reads from RDS in batches and writes to DynamoDB with BatchWriteItem. Test performance with production-scale data. Run both databases in parallel — dual writes in application code. Verify data consistency. Switch reads to DynamoDB. Stop dual writes. DynamoDB pricing is very different from RDS — model costs carefully before committing.

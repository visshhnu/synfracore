# Nginx — Interview Questions

**What is the difference between Nginx as a web server vs reverse proxy?**
As a web server, Nginx serves static files (HTML, CSS, JS, images) directly from disk — extremely efficient, handles thousands of concurrent connections with minimal memory. As a reverse proxy, Nginx forwards requests to backend servers (Node.js, Python, Java, etc.) and returns the response. The backend handles dynamic content; Nginx handles SSL termination, load balancing, caching, compression, and connection management. In production: almost always both — Nginx serves static files directly (much faster than asking the app server) and proxies dynamic requests to the backend. This hybrid approach significantly reduces backend load.

**How does Nginx handle concurrency? Explain the event-driven model.**
Apache (traditional) uses a thread/process per connection — 1000 concurrent connections = 1000 threads, each consuming ~2MB RAM = 2GB just for threads. Nginx uses a single-threaded event loop (per worker process) with non-blocking I/O — one worker handles thousands of concurrent connections. When a connection is waiting for disk I/O or upstream response, the worker handles other connections instead of blocking. Worker processes match CPU cores (worker_processes auto). Result: Nginx handles 10,000+ concurrent connections per worker with ~2MB memory vs Apache's 2GB for 1000 connections.

**What is the difference between location blocks and server blocks?**
Server blocks define virtual hosts — multiple domains on one Nginx instance. Each `server {}` block specifies a hostname (`server_name api.example.com`) and listens on a port. Nginx matches incoming requests to server blocks by hostname. Location blocks within a server block match request URI paths and define how to handle them. `location / {}` matches all requests; `location /api/ {}` matches the /api/ path; `location ~* \.(jpg|png)$ {}` uses regex matching. Nginx uses longest-prefix matching for location selection (exact match beats prefix, which beats regex in priority).

**How do you configure Nginx for zero-downtime deployments?**
Nginx's `reload` (nginx -s reload or systemctl reload nginx) loads the new configuration without dropping existing connections — old workers finish their current requests while new workers handle new connections. For upstream changes, use `upstream` directive with multiple servers and the `weight`, `down`, `backup` parameters. Blue-green: point upstream to green servers, reload. For gradual traffic shifting, use `split_clients` to route a percentage of traffic to new version. Upstream keepalive reduces connection overhead. The key: `reload` is safe; `restart` causes brief downtime.

**What is the worker_connections setting and how do you size it?**
`worker_connections` is the maximum number of simultaneous connections each Nginx worker process can handle. Total connections = worker_processes × worker_connections. A connection from client to Nginx is one connection; Nginx to upstream is another — so one client request uses 2 connections. Check `ulimit -n` (OS open file limit) — worker_connections should be ≤ this value. For a load balancer with 4 workers and worker_connections 1024: can handle ~2000 client connections (each using 2 connections). Tune based on your RAM and file descriptor limits. Use `ab` or `wrk` to load test and find the practical limit.

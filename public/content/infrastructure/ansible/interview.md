# Ansible — Interview Questions

**What is idempotency and why is it important in Ansible?**
Idempotency means running the same operation multiple times produces the same result as running it once. In Ansible, a well-written playbook can be run 10 times on the same server and will only make changes when needed — if nginx is already installed, the install task reports "ok" not "changed." This is critical for reliability: you can re-run playbooks after partial failures without creating duplicate resources or inconsistent states. Most Ansible modules are idempotent by default. The `command` and `shell` modules are NOT idempotent — use `creates` argument or `changed_when: false` to handle them correctly.

**What is the difference between command, shell, and raw modules?**
`command` runs a command without a shell — no pipes, redirects, or shell variables. Safer and faster. Use when the command doesn't need shell features. `shell` runs through `/bin/sh` — supports pipes, redirects, `&&`, `||`, environment variables. Use when you need shell functionality. `raw` sends raw SSH commands without the Python/Ansible module infrastructure — used when Python isn't installed on the target (network devices, minimal containers). For most tasks, prefer purpose-built modules (apt, service, file) over command/shell — they're idempotent and more readable.

**How does Ansible handle secrets and sensitive data?**
Ansible Vault encrypts files or individual strings with AES256. Encrypt entire files (`ansible-vault encrypt secrets.yml`) or inline strings (`ansible-vault encrypt_string 'password' --name 'db_pass'`). The vault password can be a file, a script that fetches from AWS Secrets Manager/Vault, or prompted interactively. For production: store vault password in CI/CD secrets (GitHub Actions Secret, Jenkins Credentials), never in the repo. Alternative: don't use Vault at all — fetch secrets from AWS Secrets Manager or HashiCorp Vault directly in tasks using the lookup plugin, so secrets never hit disk.

**What is Ansible inventory and what types are there?**
Inventory defines which hosts Ansible manages. Static inventory: INI or YAML file with hosts and groups — simple but requires manual updates. Dynamic inventory: a script or plugin that queries an external source (AWS EC2, Azure, GCP, Kubernetes) and returns hosts in real time. EC2 dynamic inventory automatically discovers all running instances with their metadata (tags, IPs, instance types). Dynamic inventory is essential in cloud environments where IPs change. Group variables (`group_vars/`) apply variables to all hosts in a group; host variables (`host_vars/`) apply to individual hosts.

**How do you optimize slow Ansible playbooks?**
Enable pipelining (`pipelining = True` in ansible.cfg) — reduces SSH round trips by running multiple modules in one connection. Increase forks (`forks = 20`) to run against more hosts in parallel (default is 5). Use `gather_facts: no` when you don't need system facts — saves ~1s per host. Use `--limit` to target specific hosts during development. Cache facts with `fact_caching = jsonfile` — avoids re-gathering facts on subsequent runs. Use `async` + `poll` for long-running tasks (let them run in background, check status periodically). Profile with `ANSIBLE_CALLBACK_WHITELIST=profile_tasks`.
